# Dispatch System Architecture Review

## Overall Assessment

The direction is good. The move from local SQLite-backed spawning to a
durable server/handler model is the right next step, and the docs are
clear enough that the intended control flow is easy to follow.

The main problem is that several "fundamental" invariants are not yet
fully aligned across the design docs and the current implementation. The
biggest risks are identity mismatches around sessions/codebases,
insufficient ownership recovery semantics for commands, and a command
execution model that is still too string-based for cloud mode.

## Critical Concerns

### 1. Session identity and uniqueness are inconsistent

The docs define a session as `worker_type + machine + directory`, but the
proposed schema enforces `UNIQUE (worker_type, directory)` in
`docs/active-design/dispatch-api.md`.

That creates two problems:

- It contradicts the stated session identity by excluding `machine_id`.
- It assumes `directory` is globally meaningful across machines, which it
  usually is not.

Example: `/repo/frontend` on `laptop-A` and `/repo/frontend` on `laptop-B`
would collide even if they are unrelated local paths. If the real intent
is "one worker type per codebase", then the system needs a first-class
`codebase_id` or canonical repo root identity, not a raw directory
string.

Recommendation:

- Separate `session_id` from `codebase_id`.
- Make session uniqueness something like `(worker_type, machine_id,
  canonical_directory)` if sessions are machine-local.
- If dispatch-by-codebase is a primary feature, define how codebase
  identity is discovered and normalized.

### 2. The handler state model does not match "one handler, many sessions"

The design repeatedly says one handler manages multiple sessions, but the
state file examples in `docs/active-design/dispatch-api.md` and
`docs/future/multi-agent-orchestration.md` still center on a single
`current` command.

That is not sufficient once a handler runs more than one session
concurrently. It breaks or at least leaves undefined:

- Concurrent workers on the same machine.
- Session-specific cancel operations.
- Recovery after a crash when more than one child process existed.
- Per-session sequencing guarantees.

Recommendation:

- Store state per session, not a single global `current`.
- Make command ordering and concurrency limits explicit per session.
- Treat session ownership as independent state machines under one handler.

### 3. Claim/start/finish needs leases or heartbeats

Three-phase ack is useful, but it is not enough by itself for distributed
recovery.

Failure cases still need clear ownership semantics:

- Handler claims, then dies before `start`.
- Handler starts the worker, loses network, but the worker keeps running.
- Server restarts and sees stale claimed/started commands.

Without a lease or heartbeat, the server cannot safely decide when a
claimed or started command is abandoned versus merely partitioned.

Recommendation:

- Add a renewable lease on claimed/started commands.
- Record `lease_expires_at` and refresh it from the handler.
- Add a reconciliation process that moves expired commands back to a
  recoverable state.

### 4. Raw command strings are too fragile and too weak for cloud mode

The docs say the handler is a dumb executor and the server sends a full
command string. The current implementation in
`internal/dispatch/service.go` is already showing why that is risky: it
uses `strings.Fields` to split commands before spawning.

Problems:

- Quoting and escaping are not reliable.
- Cross-platform behavior will diverge.
- Security guarantees depend on string construction discipline.
- "Whitelist by template" is weaker than "validated structured argv".

Recommendation:

- Do not send raw shell commands as the durable command payload.
- Store structured execution data: executable, argv, cwd, env allowlist,
  session ID, dispatch ID.
- For cloud mode, sign the structured payload and have the handler verify
  it before execution.

### 5. Cancel/redirect races are underspecified

The cancel and redirect flows are understandable at the narrative level,
but the state machine is not fully specified.

Missing edge cases:

- Cancel arrives before claim.
- Cancel arrives after claim but before start.
- Cancel races with normal completion.
- Redirect issues cancel + spawn, but spawn is delivered before cancel is
  fully observed by the handler.

This will produce double-finishes, leaked busy sessions, or incorrect
lineage unless the transitions are CAS-style and idempotent.

Recommendation:

- Define command and dispatch state transitions formally.
- Guarantee per-session ordering for commands.
- Make all lifecycle endpoints idempotent with explicit preconditions.
- Tie `sessions.current_dispatch_id` updates to transactional compare/set
  rules.

## Suggestions

### 1. Define directory overlap semantics

"One session per worker type per directory" is not enough until you define
what happens with:

- `/repo`
- `/repo/frontend`
- symlinks/junctions
- different path spellings pointing to the same location

Use canonicalized absolute paths at minimum. If overlap matters, enforce
it explicitly instead of only exact-string equality.

### 2. Add machine/session liveness separately

`machines.last_poll_at` is useful, but session liveness is a different
thing from machine liveness. A handler can be alive while one session is
wedged.

Consider:

- machine heartbeat
- session last_started_at / last_activity_at
- handler instance ID to distinguish restarts on the same machine

### 3. Plan the migration path from current MVP terminology

The implemented code still uses `Agent`, `Message`, and direct worker
spawn terminology in `internal/dispatch/service.go`, while the new design
is centered on `sessions`, `commands`, and handlers.

That is manageable, but the migration should be explicit:

- Which existing tables survive?
- Which concepts are renamed versus replaced?
- How will old local-only dispatches coexist with API-backed ones during
  rollout?

### 4. Capture stdout/stderr explicitly

The proposed `finish` payload only shows `exit_code` and `result`. In
practice, debugging failed workers will require:

- stdout
- stderr
- termination reason
- whether the process was canceled, killed, or exited normally

## Questions

1. Is `directory` meant to identify a local filesystem path or a logical
   codebase?
2. Can one handler run multiple worker processes concurrently, or is the
   intended concurrency actually one active process per machine?
3. In cloud mode, who is trusted to create sessions: the server, the
   handler, or a human CLI with credentials?
4. Is activity durable audit data, ephemeral telemetry, or both?
5. Are workers expected to modify only files in the session directory, and
   if so, where is that enforced?

## Alternative Approaches To Consider

### 1. Separate control plane from execution plane more strictly

Keep the current idea, but model the API as a control plane with
structured jobs and leases, not shell commands. The handler becomes an
execution adapter that translates a signed job spec into a local process.

### 2. Use a durable queue pattern instead of bespoke command lifecycle

You do not need a full external orchestrator, but borrowing established
job-runner patterns would reduce risk:

- queued -> leased -> running -> finished/failed/canceled
- lease renewal
- session-scoped ordering
- reconciliation of abandoned work

### 3. Introduce a first-class `codebase` entity

If routing by codebase is important, model it directly:

- `codebases`
- `machines`
- `sessions` attached to both

That removes a lot of ambiguity now carried by `directory`.

## Bottom Line

The architecture is viable, but I would not treat the current session
definition, uniqueness rule, and command payload design as settled. The
next design pass should tighten identity, recovery ownership, and command
execution semantics before implementation starts in earnest.
