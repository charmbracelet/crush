# Dispatch System Architecture Review

## Overall Assessment

This design is materially stronger than the earlier rounds. The move to
structured execution specs, explicit leases, per-session state, and local
result persistence addresses most of the obvious failure cases in the MVP.
The overall split between server coordination and machine-local handler
execution still looks right.

What remains is mostly about tightening invariants. The architecture now
has the right pieces, but some of the boundaries are still too permissive:
session ordering, stale ownership after lease expiry, process-tree cleanup,
and how much authority the server has over local execution details.

## Critical Concerns

### 1. Lease expiry needs fencing, not just timeout-based reclaim

The current claim/heartbeat/start/finish flow is close, but lease expiry by
itself is not enough to prevent stale handlers from writing back after a
command has been reclaimed.

Failure case:

1. Handler A claims and starts a worker.
2. Network partition prevents heartbeats.
3. Server expires the lease and reassigns the command.
4. Handler A comes back and posts `finish` or `result`.

If the server only checks command ID, the stale writer can overwrite the
new owner or incorrectly complete the dispatch.

Recommendation:

- Add a fencing token or lease generation number returned by `claim`.
- Require that token on `heartbeat`, `start`, `finish`, and result submit.
- Reject writes from older generations after reclaim.

### 2. Session-scoped ordering is still underspecified

The design says one handler manages many sessions concurrently, and
commands are polled per machine. That is fine, but the server must define
whether it can ever expose more than one actionable command for the same
session at once.

Without explicit ordering rules, these races remain:

- `spawn` and `cancel` both visible for the same session.
- `cancel` for dispatch A racing with `spawn` for redirect attempt B.
- Two handlers on the same machine restart boundary both trying to act on
  different commands for one session.

Recommendation:

- Treat each session as a single-flight lane.
- Enforce at most one leaseable command per session at a time.
- Make redirect sequencing explicit: cancel old attempt, observe terminal or
  canceled state, then release next spawn.

### 3. Process fingerprint is not enough without process-tree control

PID plus start time is a good guard against killing the wrong process, but
it does not solve orphaned child processes.

Typical CLI tools spawn subprocesses for shells, git, language servers,
watchers, or helper binaries. Killing only the top-level PID will leave
grandchildren running, holding locks, mutating files, or streaming output
after the session is supposedly free.

Recommendation:

- Start each worker in its own process group or platform-specific
  equivalent.
- On cancel, timeout, and crash recovery, terminate the full tree.
- Document the Windows approach explicitly, likely a Job Object, because
  Unix process-group assumptions do not carry over cleanly.

### 4. Structured specs still need tighter local authority boundaries

`binary + args + cwd + timeout` is much better than raw shell strings, but
the handler is still accepting a fairly powerful remote execution payload.

Open questions in the current design:

- Can the server set arbitrary `cwd`, or must it match the registered
  session directory?
- Can the server pick any binary, or only the session's registered worker
  command template?
- Will cloud-mode signature verification cover the exact argv payload, cwd,
  timeout, and environment?

Recommendation:

- Bind execution to the registered session definition.
- Treat session registration as the authority for executable and base
  directory.
- Let dispatch commands provide only bounded parameters, not arbitrary
  executable choice.
- If environment variables are introduced later, make them allowlist-only.

### 5. Result idempotency is too coarse if keyed only by dispatch ID

The docs say result submission is idempotent and keyed by dispatch ID. That
works only if there is exactly one valid execution result per dispatch and
no stale writer can race after reclaim.

Once you have:

- lease expiry and reclaim,
- retries,
- cancel racing with normal completion,
- redirect lineage,

you need to identify which execution attempt produced the accepted result.

Recommendation:

- Key execution completion by command ID plus lease generation, not just
  dispatch ID.
- Keep dispatch result materialization as a derived state from the accepted
  terminal command.

## Suggestions

### 1. Use opaque session IDs and keep names as labels

Human-readable names like `goose-frontend` are good UX, but they are not a
great primary identity if you later need renames, duplicate-friendly labels,
or stronger referential integrity. A stable opaque `session_id` plus a
display name will age better.

### 2. Canonicalize directories defensively

The canonicalization section is good, but Windows needs extra care:

- drive-letter case,
- UNC paths,
- junctions,
- symlinks that fail to resolve when the path does not exist yet.

I would document exact failure behavior for registration when the directory
cannot be fully canonicalized.

### 3. Separate durable audit data from live activity transport

`POST /dispatch/{id}/activity` currently reads like both telemetry and audit
log ingestion. Those are different retention and failure semantics.

Recommendation:

- Decide whether activity is best-effort live streaming, durable event log,
  or both.
- If both, model them separately now so the API shape does not get muddled.

### 4. Add explicit concurrency and backpressure limits

One handler can manage many sessions, but the docs do not yet say:

- max concurrent processes per machine,
- max buffered activity per session,
- max pending results on disk,
- what happens when the machine is overloaded.

Those limits will matter quickly in real use.

## Questions

1. Is a reclaimed command expected to restart the same dispatch attempt, or
   create a new execution attempt record under the same dispatch?
2. When a session is busy and a second dispatch targets it, is the intent to
   queue per session on the server or reject immediately?
3. Are handlers allowed to register or mutate sessions in cloud mode, or is
   session registration an administrative action only?
4. Is `activity` intended to include stdout/stderr verbatim, structured
   status messages, or both?
5. Do you want session directories to represent arbitrary paths, or should
   they eventually map to a first-class codebase/repository identity?

## Alternative Approaches We Should Consider

### 1. Treat the command table as a real job-runner queue

You do not need Temporal or Kubernetes for this, but you should borrow the
stronger patterns from durable job systems:

- per-lane ordering,
- leases with fencing,
- terminal event derivation,
- reconciler for abandoned executions.

That keeps the current architecture while reducing bespoke edge cases.

### 2. Introduce an explicit execution-attempt record

Right now dispatch, command, and result are doing a lot of overlapping work.
An explicit `execution_attempts` table could make recovery and idempotency
cleaner:

- dispatch = user intent,
- command = instruction delivered to handler,
- execution attempt = actual owned process lifecycle.

### 3. Consider repository identity separately from directory identity

If this system expands beyond one machine or one checkout per repo,
`directory` will eventually stop being enough to express routing intent.
Not required for phase 1, but worth keeping space for in the schema.

## Bottom Line

This is now a viable implementation plan. I would proceed, but only after
locking down three things in the spec: lease fencing, session-scoped command
ordering, and process-tree termination semantics. Those are the main places
where a system that looks reliable on paper still fails under real crash and
partition behavior.
