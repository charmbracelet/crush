# Dispatch System Architecture Review

## Overall Assessment

The core direction is sound. Session-scoped execution, structured specs, leases,
and per-session recovery state are the right primitives for moving from the MVP
to a distributed system.

The main issue is that a few of those primitives still conflict with each other
in the current design. The largest problems are around cancellation, lease
expiry while a process is still running, and what exactly is authoritative when
an execution is reclaimed.

There is also some doc drift between the current implementation and the docs:
the implemented service in `internal/dispatch/service.go` still shells out from
`cli_command` using `strings.Fields`, appends an alert string, and does not yet
match the API-based worker flow described in the review materials. That is fine
for an MVP, but the review docs should keep emphasizing that this is not just
"not implemented yet"; it is a materially different trust and execution model.

## Critical Concerns

### 1. Cancel-as-command conflicts with one-command-per-session ordering

This is the biggest design contradiction.

You say:

- only one actionable command per session at a time
- the next command is blocked until the current one is terminal
- cancel is modeled as another command in the same queue

Those cannot all be true simultaneously. If `spawn` for a session is already in
`started`, a `cancel` command for that same session cannot be delivered until
the spawn becomes terminal, but the whole purpose of cancel is to make it
terminal.

You need one of these models:

- cancellation is an out-of-band mutation on the active command, not a queued
  command, or
- cancellation is allowed to preempt the active command inside the same session
  state machine, with explicit precedence rules.

Without that change, cancel/redirect is underspecified and will deadlock.

### 2. Lease expiry can cause duplicate execution after network partition

The design currently says:

- handler claims and starts a process
- lease expires if heartbeats stop
- server returns the command to `pending`
- another handler may reclaim it

If the original process is still running locally, you now have two workers that
believe they own the same dispatch attempt.

Fencing protects the database writes, but it does not protect the side effects
inside the repo or external systems. A stale worker can still edit files, push a
branch, open PRs, or call APIs after its lease is lost.

That means "started -> pending on lease expiry" is too aggressive. Once a
process has been started, lease loss should move the command into a recovery
state that requires machine-local reconciliation before re-execution, not
immediate requeue.

### 3. Fencing is applied to command/result writes, but not to execution side effects

Related to the previous point: `(command_id, lease_generation)` is sufficient
for idempotent API writes, but not sufficient for correctness of the actual
work. If a worker keeps running after its lease expires, the stale generation
prevents a late `finish`, but it does not prevent stale code changes.

You need an explicit answer to: what does the system guarantee when a started
worker becomes partitioned? Right now the docs imply stronger safety than the
mechanics actually provide.

At minimum, I would document the guarantee as:

- fencing prevents stale state transitions and stale result submission
- fencing does not prevent stale in-repo side effects from a partitioned worker

Then design mitigation around that limitation.

### 4. Result persistence is keyed inconsistently across the design

The review docs correctly say result idempotency should key on
`(command_id, lease_generation)`, but the sample per-session state file stores
`pending_results` keyed by `dispatch_id`-shaped data and example retry flow is
phrased in dispatch terms.

That is risky because a single dispatch may have:

- multiple command attempts after lease expiry
- redirect lineage
- cancel/respawn sequences

The durable local retry unit should be "command attempt result", not just
"dispatch result". Otherwise stale result replay becomes ambiguous.

I would store, at minimum:

- `command_id`
- `lease_generation`
- `dispatch_id`
- terminal command metadata
- result submission acknowledgement status

### 5. Spec authority validation is too weak if the server is not fully trusted

Validating only `binary` and exact `cwd` is not enough to call the spec
"authorized". The server still controls:

- arbitrary `args`
- arbitrary environment variables
- timeout
- potentially output capture behavior

If the server is trusted, this is acceptable and should be stated plainly. If
the handler is intended to enforce a machine-local policy boundary, it does not
yet do that.

A stronger model would register a session execution profile that constrains:

- allowed binary identity
- allowed fixed prefix arguments
- allowed/denied environment keys
- max timeout
- maybe a small set of allowed verbs per worker type

### 6. Directory conflict rules are path-correctness sensitive on Windows

The directory model is reasonable, but the overlap check is more fragile than
the docs suggest. Windows adds case-insensitivity, UNC paths, junctions, 8.3
paths, drive-letter aliases, and mixed separator forms. `EvalSymlinks` plus
`filepath.Rel` is necessary but probably not sufficient for all collisions.

If the design depends on directory identity for corruption prevention, the
canonicalization and overlap rules need to be treated as a security boundary,
not just a convenience helper.

## Suggestions

### 1. Split command state from dispatch state more explicitly

The docs blur them in a few places. I would define:

- dispatch: user-visible logical task
- command: one concrete execution attempt against a session
- result: terminal artifact of one command attempt

That makes retries, redirect lineage, and cancellation much easier to reason
about.

### 2. Add a `cancel_requested` or `interrupt_requested` bit on the active command

This avoids the queued-cancel contradiction. The handler can observe the bit on
heartbeat renewal or via a direct cancel endpoint and terminate the running
process without needing a second actionable command.

### 3. Add a recovery state for started commands that lose their lease

Suggested flow:

- `pending -> claimed -> started`
- if lease expires before finish, transition to `orphaned` or `recovery_needed`
- require handler restart/reconciliation/explicit operator action before retry

That is safer than automatic requeue after start.

### 4. Make lease duration adaptive or at least configurable per worker type

`60s/30s` is a reasonable default, but some CLIs block for long periods during
startup or tool execution. Make it configurable and add jitter so multiple
sessions do not synchronize their heartbeats.

### 5. Treat process-tree termination as platform-specific from the beginning

The Unix `Setpgid` approach is right. On Windows, "Phase 2 Job Objects" is not
really optional if the safety story depends on killing descendants. I would mark
Windows support as incomplete until Job Objects are present.

### 6. Add an explicit machine incarnation / session incarnation concept

You already have `handler_instance_id` in the schema. Lean on it harder. A
claim/started command should be tied to both:

- machine identity
- handler incarnation on that machine

That will make recovery and stale ownership easier to reason about.

## Questions

1. What is the exact server behavior when a `started` command's lease expires
   but the machine is still polling for other sessions?

2. Is cancel intended to be best-effort only, or should it provide a strong
   guarantee that no further worker side effects occur after acknowledgement?

3. Can a single dispatch ever intentionally produce more than one result
   document, or is there always exactly one terminal result per dispatch
   attempt?

4. For cloud mode, is the server trusted to choose arbitrary worker arguments,
   or is the handler supposed to enforce a local execution policy?

5. How will you identify the correct process start time on macOS and Windows in
   a way that is stable enough for crash recovery?

## Alternative Approaches To Consider

### 1. Active-command row instead of queued cancel commands

Model one active execution row per session and mutate it with:

- `cancel_requested`
- `redirect_requested`
- `superseded_by`

That fits the session-serialized model better than a separate command queue for
control-plane actions.

### 2. Use two levels of retry semantics

- dispatch retry: human-meaningful new attempt
- command retry: transport/recovery retry of the same attempt

That separation will reduce confusion around lineage and result idempotency.

### 3. Consider "at-most-one live process per session" as the primary invariant

Right now the docs emphasize queue ordering and lease ownership. The more useful
invariant is probably:

- there must never be two live worker processes for the same session

Then shape claim expiry, recovery, and cancel semantics around preserving that
invariant first.

## Bottom Line

The architecture is close, but I would not implement it exactly as written.

Before building the API, I would first resolve:

1. how cancel works without violating session ordering,
2. what happens after lease loss for already-started processes, and
3. whether handler-side spec validation is a real policy boundary or just a
   sanity check.

Those three choices will determine whether the rest of the system is merely
operationally awkward or actually unsafe under failure.
