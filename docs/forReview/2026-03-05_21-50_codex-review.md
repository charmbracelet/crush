# Dispatch System Architecture Review

## Overall Assessment

This revision is materially stronger than the last one. The two biggest issues
from the prior round are fixed: cancel is now out-of-band, and lease expiry for
started work moves to `orphaned` instead of requeueing immediately.

What remains is mostly internal consistency. The high-level model is now
coherent, but several examples and pseudo-implementations still describe the old
queued-cancel model or have races that would break the handler if implemented
literally.

## Critical Concerns

### 1. The docs still partially model cancel as a queued command

The design now says cancel is out-of-band and not a queued command, which is the
right change. But the implementation sections still contradict that:

- `commands.action` still lists `cancel` as a command type
- `processCommand` still switches on `cmd.Action == "cancel"`
- the ordering section still says "Cancel always follows spawn for same dispatch"
- implementation tasks still say "Implement cancel command"

If someone codes from those sections instead of the narrative sections, they can
reintroduce the deadlock you just removed.

### 2. Cancel handling races with the worker wait path and can double-finish

The handler heartbeat example kills the process and immediately calls
`finishCommand(...)` when `cancel_requested` is seen. But `spawnWorker()` is also
still blocked in `execCmd.Wait()` and will later prepare a result and call
`finishCommand(...)` again.

You describe `finish` as idempotent, which protects the server, but the local
handler state is still underspecified:

- which goroutine owns clearing `current_command`
- which goroutine persists `pending_results`
- how duplicate result submission is suppressed locally
- how cancel vs timeout vs normal exit is resolved deterministically

Without a single terminalization path, the handler will be racy even if the API
is idempotent.

### 3. The pseudo-code still uses dispatch-scoped pending results in places

The prose was updated to key pending results by `(command_id, lease_generation)`,
which is correct. But the pseudo-code still persists and retries by dispatch:

- `session.AddPendingResult(cmd.DispatchID, result)`
- `h.submitResult(ctx, cmd.DispatchID, result)`
- recovery retries `pending.DispatchID`

That is exactly the ambiguity the updated docs are trying to remove. If the docs
intend command-attempt identity, the handler examples need to carry
`command_id` and `lease_generation` end-to-end.

### 4. `orphaned` breaks the stated session ordering unless you define its scheduling semantics

The state machine now moves started commands to `orphaned`, but the ordering rule
only blocks sessions with commands in `pending/claimed/started`. `orphaned` is
excluded from that list while also being described as requiring manual
reconciliation.

Those statements need to be reconciled. Either:

- `orphaned` blocks new work for the session until resolved, or
- the session can accept new work while an orphan exists.

The second option reintroduces the same duplicate-side-effect risk you just
avoided. I think `orphaned` needs to make the session unavailable.

## Suggestions

### 1. Define one terminalization owner in the handler

Make one goroutine responsible for finalizing a command locally. Heartbeat can
request interruption, but the wait/monitor path should be the only place that:

- records terminal reason
- persists the pending result
- calls `finish`
- clears session state

That will eliminate most race conditions.

### 2. Remove queued-cancel remnants from the docs now

I would update all of these to avoid backsliding:

- `commands.action` examples
- `processCommand` switch example
- ordering guarantees text
- implementation checklist item "Implement cancel command"

### 3. Define session state explicitly for orphaned commands

Document one of these rules directly:

- `orphaned` commands pin the session in an unavailable state until reconciled, or
- reconciliation auto-runs on handler restart before the session becomes available.

Right now that rule is implied but not written tightly enough.

### 4. Tighten the command attempt object in state files and pseudo-code

All local persistence examples should carry the same identity tuple:

- `command_id`
- `lease_generation`
- `dispatch_id`
- terminal reason
- submission status / attempts

That makes the docs internally self-consistent.

## Questions

1. When a command becomes `orphaned`, does the session status change from `busy`
   to a third state, or does `busy` cover both active and blocked-orphaned?

2. On cancel, should the handler submit a dispatch result payload at all, or is
   `POST /commands/{id}/finish` alone sufficient to surface the cancellation?

3. Is redirect allowed while the old command is already `orphaned`, or must that
   be reconciled manually first?

## Alternative Approaches To Consider

### 1. Separate command terminal state from dispatch result publication

A small local state machine like `running -> interrupting -> exited -> reported`
would make the handler easier to reason about than folding everything into one
`finish` call.

### 2. Make session availability derived, not stored

Instead of `available/busy` alone, derive availability from:

- no active command
- no orphaned command awaiting reconciliation
- no registration conflict

That would make the session scheduler harder to mis-specify.

## Bottom Line

The design is now close. I would move forward after one cleanup pass that makes
all handler examples and schema examples match the new cancel/orphan model.

The remaining risk is not the architecture anymore; it is that the docs still
contain enough stale pseudo-code to let an implementation drift back into the
old failure mode.
