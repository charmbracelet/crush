# Dispatch System Architecture Review

## Overall Assessment

This is close. The active design now reads like a coherent system, and the last
round's main correctness problems appear resolved in that document.

What remains is smaller but still important: the state model exposed to clients
is still underspecified, and the future-vision doc has not been fully brought
forward to match the active design.

## Critical Concerns

### 1. Session status is still too coarse for the new orphaned model

The active design still says session status is only `available` or `busy`, and
`GET /sessions` still filters only by those two states. But the command model now
has `orphaned`, and the scheduling rule correctly blocks sessions with orphaned
commands from receiving more work.

That means the externally visible session state is missing an important case:
there is a difference between:

- actively executing work, and
- blocked because an orphaned command requires reconciliation.

If both collapse to `busy`, operators and automation cannot tell whether the
session is healthy-but-occupied or unhealthy-and-stuck.

I would add an explicit session state such as `blocked`, `orphaned`, or
`needs_recovery`, or make it explicit that session status is derived from both
session metadata and current command state and expose that derived state in the
API.

### 2. The future vision doc still contradicts the active design

`docs/future/multi-agent-orchestration.md` still describes the old model in a
few places:

- `Command` is defined as `spawn, cancel`
- the flow still shows cancel immediately calling `FINISH`
- the commands schema still lists `action TEXT NOT NULL  -- spawn, cancel`

That directly conflicts with the active design, where cancel is out-of-band and
not a queued command.

This is no longer just wording drift. Once two design docs disagree on the
control-plane model, implementation can fork depending on which document someone
reads first.

## Suggestions

### 1. Promote orphaned/recovery to a first-class session-facing concept

The system now has a real distinction between schedulable and unschedulable
sessions that are not merely "busy". Surface that distinction in:

- `GET /sessions`
- session registration/list docs
- operator troubleshooting guidance

### 2. Update the future vision doc immediately after the active design changes

At this point `future/multi-agent-orchestration.md` should inherit the same:

- out-of-band cancel model
- single terminalization owner idea
- command-attempt result identity
- orphaned-blocks-session rule

Otherwise it is acting as a stale fork of the design.

### 3. Rename the implementation checklist item

`Implement cancel command` in the active-design checklist still reflects the old
model. It should be something like `Implement out-of-band cancel signaling` or
`Implement cancel_requested flow`.

## Questions

1. Do you want `orphaned` to be visible as a session status to clients, or only
   as an internal command state?

2. Is the future document intended to be normative, or is it just a looser
   vision doc? Right now readers could reasonably treat it as normative because
   it still contains schema and flow details.

## Alternative Approaches To Consider

### 1. Expose derived session health separately from status

If you want to keep `available/busy` for simplicity, add another field like:

- `health: ok | needs_recovery`
- `schedulable: true | false`

That avoids overloading one enum with two concerns.

## Bottom Line

I would move forward with the active design after one more cleanup pass.

The remaining issue is no longer architectural uncertainty. It is consistency:
make the session API reflect the new orphaned semantics, and make the future
vision doc stop describing the old cancel model.
